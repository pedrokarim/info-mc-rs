Errors & Rate Limiting

All API errors follow a consistent JSON format. This page documents the error structure, error codes, rate limiting rules, and cache behavior.

Error Format

Every error response contains an error object with a machine-readable code and a human-readable message:

{
  "error": {
    "code": "invalid_address",
    "message": "The server address 'not a valid address!!!' is not a valid hostname or IP address."
  }
}
Tip: Always check the error.code field for programmatic error handling. The message field is intended for display to end users and may change between versions.

Error Codes

Code HTTP Status Description
invalid_address 400 The provided server address, username, or UUID format is invalid.
dns_resolution_failed 422 The domain name could not be resolved via DNS. The server may not exist or DNS may be misconfigured.
connection_refused 502 A TCP connection to the server was actively refused. The server may be down or firewalled.
timeout 504 The server did not respond within the timeout period (default: 5 seconds).
player_not_found 404 No Minecraft account exists with the given username or UUID.
rate_limit_exceeded 429 You have exceeded the rate limit for this endpoint. Wait and retry after the reset time.
internal_error 500 An unexpected internal server error occurred. If this persists, please report it.

Rate Limiting

Rate limits are applied per IP address on a sliding window basis. Different endpoint groups have different limits:

Endpoint Group Rate Limit Window
Server endpoints (/api/v1/server/*) 30 requests per minute
Player endpoints (/api/v1/player/*) 60 requests per minute
Render endpoints (/api/v1/render/*) 60 requests per minute

Rate Limit Headers

Every response includes rate limit headers so you can track your usage:

Header Description
X-RateLimit-Limit Maximum number of requests allowed in the current window.
X-RateLimit-Remaining Number of requests remaining in the current window.
X-RateLimit-Reset Unix timestamp (in seconds) when the rate limit window resets.

429 Response Example

{
  "error": {
    "code": "rate_limit_exceeded",
    "message": "Rate limit exceeded. Try again in 34 seconds."
  }
}
# Response headers on a 429:
HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 30
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1741789834
Retry-After: 34
Important: Continuing to send requests after receiving a 429 response will not reset the window. Respect the Retry-After header or X-RateLimit-Reset timestamp to avoid further rate limiting.

Cache Behavior

Responses are cached server-side to reduce latency and load on upstream services. The cached field in JSON responses indicates whether the data was served from cache.

Data Type Cache TTL
Server ping 60 seconds
Player profile 5 minutes
Skin PNG 1 hour
Skin render 1 hour
DNS SRV lookup 5 minutes
Tip: Cached responses do not count against your rate limit. You can check the cached field in the JSON response to know if data was served from cache.
