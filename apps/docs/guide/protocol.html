Protocol Implementation

The mc-protocol crate implements the Minecraft network protocol from scratch. It supports three distinct ping methods: Server List Ping (Java Edition, TCP), Query Protocol (Java Edition, UDP), and Bedrock Raknet Ping (Bedrock Edition, UDP).

VarInt Encoding

Minecraft uses a variable-length integer encoding for most numeric fields in its protocol. A VarInt uses between 1 and 5 bytes, where each byte uses 7 bits for data and 1 bit as a continuation flag.

Encoding

/// Encode a 32-bit integer as a Minecraft VarInt
pub fn write_varint(mut value: i32) -> Vec<u8> {
    let mut buf = Vec::new();
    loop {
        let mut byte = (value & 0x7F) as u8;
        value = ((value as u32) >> 7) as i32;
        if value != 0 {
            byte |= 0x80;
        }
        buf.push(byte);
        if value == 0 {
            break;
        }
    }
    buf
}

Decoding

/// Decode a Minecraft VarInt from a byte stream
pub async fn read_varint(stream: &mut (impl AsyncRead + Unpin)) -> Result<i32> {
    let mut result: i32 = 0;
    let mut shift: u32 = 0;
    loop {
        let mut buf = [0u8; 1];
        stream.read_exact(&mut buf).await?;
        let byte = buf[0];
        result |= ((byte & 0x7F) as i32) << shift;
        if byte & 0x80 == 0 {
            break;
        }
        shift += 7;
        if shift >= 32 {
            return Err(anyhow!("VarInt too large"));
        }
    }
    Ok(result)
}

VarLong

VarLong uses the same encoding as VarInt but supports 64-bit values, requiring up to 10 bytes. The algorithm is identical with a shift limit of 64 instead of 32.

MC String

Minecraft strings are length-prefixed using a VarInt, followed by UTF-8 encoded bytes:

/// Write a Minecraft protocol string
pub fn write_string(s: &str) -> Vec<u8> {
    let mut buf = write_varint(s.len() as i32);
    buf.extend_from_slice(s.as_bytes());
    buf
}

MC Packet Format

Every Minecraft packet follows the same structure:

Field Type Description
Length VarInt Total byte length of Packet ID + Data
Packet ID VarInt Identifies the packet type
Data byte[] Packet-specific payload

Server List Ping (SLP)

The Server List Ping is a TCP-based protocol used by Minecraft Java Edition clients to fetch server information from the multiplayer menu.

TCP Flow

  Client                          Server
    │                               │
    │──── Handshake (0x00) ────────▶│
    │     - Protocol version        │
    │     - Server address          │
    │     - Server port             │
    │     - Next state = 1          │
    │                               │
    │──── Status Request (0x00) ───▶│
    │                               │
    │◀─── Status Response (0x00) ───│
    │     - JSON payload            │
    │                               │
    │──── Ping (0x01) ────────────▶│
    │     - Timestamp (i64)         │
    │                               │
    │◀─── Pong (0x01) ─────────────│
    │     - Same timestamp          │
    │                               │

JSON Response Format

The server responds with a JSON object containing server information:

{

Query Protocol

The Query protocol is a UDP-based protocol that provides more detailed server information. It must be explicitly enabled in server.properties with enable-query=true.

Handshake

Before querying, the client must perform a handshake to obtain a challenge token:

Step Direction Packet
1 Client → Server Handshake request (type=0x09, session ID)
2 Server → Client Challenge token (ASCII number string)

Basic Stat

A basic stat query returns minimal information: MOTD, game type, map, current players, max players, host port, and host IP.

Full Stat

A full stat query returns comprehensive information including all key-value pairs and a full player list. The response uses a padding scheme with splitnum\x00\x80\x00 as a delimiter between the KV section and the player list.

Plugin Parsing

The plugins field in the full stat response follows a specific format:

# Format: "ServerSoftware: Plugin1 vX.Y; Plugin2 vA.B"
# Example:
"Paper 1.21.4: EssentialsX 2.20.1; WorldEdit 7.3.0; ViaVersion 4.9.0"

The parser splits on : to separate the server software, then splits the remainder on ; to extract individual plugins.

Bedrock Raknet Ping

Bedrock Edition uses the Raknet protocol over UDP. MCInfo sends an Unconnected Ping packet and receives an Unconnected Pong containing a server ID string.

Server ID String Format

DNS SRV Resolution

Before connecting to a Minecraft server, the client should check for a DNS SRV record at _minecraft._tcp.{hostname}. This allows server operators to run Minecraft on non-standard ports without requiring users to specify the port.

Resolution Algorithm

  1. If the address contains a port (e.g., play.example.com:25565), use it directly
  2. Otherwise, perform an SRV lookup for _minecraft._tcp.{hostname}
  3. If an SRV record exists, use the target host and port from the record
  4. If no SRV record exists, fall back to the hostname with the default port (25565 for Java, 19132 for Bedrock)
/// Resolve a Minecraft server address using DNS SRV records
pub async fn resolve_address(address: &str) -> (String, u16) {
    // Check if port is already specified
    if let Some((host, port)) = address.rsplit_once(':') {
        if let Ok(port) = port.parse::<u16>() {
            return (host.to_string(), port);
        }
    }

    // Try SRV lookup
    let srv_name = format!("_minecraft._tcp.{address}");
    if let Ok(records) = resolve_srv(&srv_name).await {
        if let Some(record) = records.first() {
            return (record.target.clone(), record.port);
        }
    }

    // Fall back to default port
    (address.to_string(), 25565)
}

Java vs Bedrock Detection

MCInfo can automatically detect whether a server is Java or Bedrock edition by running both ping methods in parallel using Tokio:

/// Ping a server, auto-detecting Java vs Bedrock
pub async fn ping_auto(address: &str) -> Result<ServerInfo> {
    let java = tokio::spawn(ping_java(address.to_string()));
    let bedrock = tokio::spawn(ping_bedrock(address.to_string()));

    // Return whichever succeeds first, or both if available
    let (java_result, bedrock_result) = tokio::join!(java, bedrock);

    match (java_result?, bedrock_result?) {
        (Ok(j), Ok(b)) => Ok(ServerInfo::Both { java: j, bedrock: b }),
        (Ok(j), Err(_)) => Ok(ServerInfo::Java(j)),
        (Err(_), Ok(b)) => Ok(ServerInfo::Bedrock(b)),
        (Err(e), Err(_)) => Err(e),
    }
}

Error Handling

The protocol crate uses typed errors to distinguish between different failure modes:

Error Cause Action
ConnectionTimeout Server did not respond within the timeout window Server may be offline or firewalled
ConnectionRefused TCP connection actively refused Port is not open or server is not running
InvalidPacket Response could not be parsed Protocol mismatch or corrupted data
DnsResolutionFailed Hostname could not be resolved Invalid hostname or DNS issue
VarIntTooLarge VarInt exceeded 5 bytes Malformed packet data
UnexpectedEof Stream ended before expected data Connection was closed prematurely
