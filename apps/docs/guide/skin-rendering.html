Skin Rendering

MCInfo provides both 2D and 3D skin rendering capabilities. The mc-skin crate handles 2D face/body rendering, while mc-render3d produces full 3D renders using wgpu for headless GPU rendering on the server.

3D Model Architecture

The Minecraft player model is composed of six cuboid body parts, each positioned relative to the model origin. All dimensions are in Minecraft pixel units (1 unit = 1 pixel on the skin texture).

Part Size (W x H x D) Position (X, Y, Z) Pivot
Head 8 x 8 x 8 (0, 24, 0) Center-bottom
Torso 8 x 12 x 4 (0, 12, 0) Center
Right Arm 4 x 12 x 4 (classic) / 3 x 12 x 4 (slim) (-6, 12, 0) / (-5.5, 12, 0) Top-center
Left Arm 4 x 12 x 4 (classic) / 3 x 12 x 4 (slim) (6, 12, 0) / (5.5, 12, 0) Top-center
Right Leg 4 x 12 x 4 (-2, 0, 0) Top-center
Left Leg 4 x 12 x 4 (2, 0, 0) Top-center
Cape 10 x 16 x 1 (0, 24, -2) Top-center

Overlay Layers

Each body part has an optional overlay layer (hat, jacket, sleeves, pants) that renders slightly larger than the base layer to prevent z-fighting. The overlay uses an epsilon offset on each face.

Overlay Base Part Epsilon Texture Region
Hat Head 0.5 px (32, 0) 32x16
Jacket Torso 0.25 px (16, 32) 24x16
Right Sleeve Right Arm 0.25 px (40, 32) 16x16
Left Sleeve Left Arm 0.25 px (48, 48) 16x16
Right Pant Right Leg 0.25 px (0, 32) 16x16
Left Pant Left Leg 0.25 px (0, 48) 16x16

UV Mapping

Each face of a cuboid is mapped to a specific region of the 64x64 skin texture. The UV coordinates are defined in pixel units and then normalized to [0, 1] for the GPU.

For a cuboid with skin texture origin at (u, v) and dimensions (w, h, d), the face UV regions are:

Face UV Origin (x, y) UV Size (w, h)
Front (u + d, v + d) (w, h)
Back (u + d + w + d, v + d) (w, h)
Right (u, v + d) (d, h)
Left (u + d + w, v + d) (d, h)
Top (u + d, v) (w, d)
Bottom (u + d + w, v) (w, d)

Example: Head UV Coordinates

The head base layer has skin origin (0, 0) with dimensions 8 x 8 x 8:

// Head base UV regions (in pixels)
// Front: (8, 8)   size 8x8
// Back:  (24, 8)  size 8x8
// Right: (0, 8)   size 8x8
// Left:  (16, 8)  size 8x8
// Top:   (8, 0)   size 8x8
// Bottom:(16, 0)  size 8x8

Slim vs Classic Detection

Minecraft skins come in two arm width variants:

  • Classic (Steve) — 4-pixel wide arms
  • Slim (Alex) — 3-pixel wide arms

The skin model type is typically provided by the Mojang API in the profile texture metadata. When unavailable, MCInfo falls back to a heuristic: checking if the rightmost column of the right arm region contains only fully transparent pixels, which indicates a slim skin.

Animation (Walking Pose)

The 3D renderer supports a walking pose that applies sinusoidal rotations to the arms and legs around their pivot points.

Swing Formulas

// Animation parameter: t (0.0 to 1.0, phase of the walk cycle)
let swing_angle = 30.0_f32.to_radians(); // Maximum swing: 30 degrees

// Arms swing opposite to the corresponding leg
let right_arm_rotation = swing_angle * (t * 2.0 * PI).sin();
let left_arm_rotation  = -right_arm_rotation;

// Legs swing in opposition
let right_leg_rotation = -right_arm_rotation;
let left_leg_rotation  = right_arm_rotation;

The rotation is applied around the X axis at each limb's pivot point (top-center), producing a natural forward/backward swing.

Interactive Rotation

The 3D viewer supports orbital camera rotation controlled by two angles:

  • Theta — horizontal rotation around the Y axis (yaw)
  • Phi — vertical rotation from the XZ plane (pitch), clamped to avoid gimbal lock
// Camera position from spherical coordinates
let camera_x = distance * phi.cos() * theta.sin();
let camera_y = distance * phi.sin();
let camera_z = distance * phi.cos() * theta.cos();

// Camera always looks at the model center
let view = Mat4::look_at_rh(
    Vec3::new(camera_x, camera_y, camera_z),
    Vec3::new(0.0, 12.0, 0.0), // Model center (half height)
    Vec3::UP,
);

Lighting and Camera Setup

The renderer uses a simple directional lighting model to add depth to the 3D render:

  • Ambient light: 0.6 intensity, applied uniformly
  • Directional light: 0.4 intensity, from upper-front-right direction
  • Camera: perspective projection with a narrow FOV to match NameMC-style framing

The lighting calculation is performed per-fragment in the shader:

// WGSL shader pseudocode
let ambient = 0.6;
let diffuse = max(dot(normal, light_dir), 0.0) * 0.4;
let final_color = texture_color * (ambient + diffuse);

Per-Pixel Face Coloring

Rather than using traditional UV-mapped textures on the GPU, mc-render3d employs a per-pixel face coloring technique. Each face of each cuboid is rasterized and the skin texture is sampled directly in Rust code, producing a vertex-colored mesh. This approach:

  • Avoids texture filtering artifacts (nearest-neighbor sampling is guaranteed)
  • Handles overlay transparency correctly by pre-compositing layers
  • Makes the wgpu pipeline simpler (no texture bind groups needed)

Cape Rendering

Capes use a separate texture that can be either 64x32 or a larger resolution that is scaled down. The cape is rendered as a flat cuboid attached to the back of the torso.

Cape Scale Detection

Cape textures from the Mojang API can come in different resolutions. The renderer detects the scale factor by dividing the texture width by 64 (the base cape texture width) and adjusts UV coordinates accordingly:

let scale = cape_texture.width() / 64;
// All UV coordinates are multiplied by `scale`
// to sample from the correct positions in the texture

Server-Side Rendering with wgpu

The mc-render3d crate uses wgpu for headless GPU rendering, allowing the API server to produce 3D skin renders without a display. The rendering pipeline:

  1. Initialize a wgpu device and queue (headless, no surface)
  2. Build the vertex/index buffers for all body parts
  3. Set up the render pipeline with depth testing enabled
  4. Render to an offscreen texture
  5. Read back the pixels and encode as PNG
// Simplified rendering flow
let (device, queue) = adapter.request_device(&default()).await?;

let texture = device.create_texture(&TextureDescriptor {
    size: Extent3d { width: 300, height: 400, depth_or_array_layers: 1 },
    format: TextureFormat::Rgba8UnormSrgb,
    usage: TextureUsages::RENDER_ATTACHMENT | TextureUsages::COPY_SRC,
    ..default()
});

// ... build mesh, encode commands, submit ...

// Read back the rendered image
let buffer = device.create_buffer(&BufferDescriptor {
    size: (300 * 400 * 4) as u64,
    usage: BufferUsages::COPY_DST | BufferUsages::MAP_READ,
    ..default()
});

encoder.copy_texture_to_buffer(/* ... */);
queue.submit([encoder.finish()]);
