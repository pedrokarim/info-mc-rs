Architecture

MCInfo-RS is designed as a self-contained tier service that talks directly to Minecraft servers and the Mojang API. It does not rely on any third-party Minecraft APIs — all data is fetched and processed in-house.

Vision

The project aims to provide a complete Minecraft server and player information service with the following principles:

  • Direct communication — query Minecraft servers using the native protocol (SLP, Query, Raknet)
  • Self-contained rendering — generate 2D and 3D skin renders server-side using wgpu
  • Aggressive caching — minimize external requests with an in-memory cache layer
  • Modular design — each concern lives in its own crate, enabling reuse and testability

System Overview

The following diagram shows how the major components interact:

                    ┌───────────────────┐
                    │   SvelteKit Web   │
                    └────────┬──────────┘
                             │ REST API
                    ┌────────▼──────────┐
                    │   mc-api (Axum)   │
                    └────────┬──────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
     ┌────────▼───┐  ┌──────▼──────┐  ┌────▼────────┐
     │ mc-protocol │  │  mc-mojang  │  │  mc-cache   │
     └─────────────┘  └─────────────┘  └─────────────┘

The SvelteKit frontend communicates with the Axum-based REST API, which orchestrates calls to the protocol layer, the Mojang API client, and the cache. Skin rendering crates (mc-skin and mc-render3d) are invoked on demand when the API needs to produce images.

Tech Stack

Layer Technology Purpose
Web Framework Axum Async HTTP server with tower middleware
Async Runtime Tokio Multi-threaded async executor for all I/O
Frontend SvelteKit SSR/SPA web application
3D Viewer (client) Three.js Interactive WebGL skin viewer in the browser
3D Rendering (server) wgpu GPU-accelerated headless skin rendering
Caching moka High-performance concurrent cache
Image Processing image PNG encoding/decoding and pixel manipulation
Serialization serde / serde_json JSON serialization for API responses

Crate Dependency Graph

The workspace is organized so that lower-level crates have no knowledge of higher-level ones. Here is the dependency graph:

  mc-api
  ├── mc-protocol      (server pinging)
  ├── mc-mojang        (Mojang API)
  ├── mc-cache         (caching layer)
  ├── mc-motd          (MOTD parsing)
  ├── mc-skin          (2D skin rendering)
  └── mc-render3d      (3D skin rendering)
      └── mc-skin      (shared skin utilities)

  mc-mojang
  └── mc-cache         (cache Mojang responses)

  mc-skin
  └── (image crate)

  mc-render3d
  ├── mc-skin          (skin texture loading)
  └── wgpu             (GPU rendering)

Key Design Decisions

  • mc-protocol is fully standalone — it only depends on tokio for async I/O and implements the Minecraft protocol from scratch
  • mc-cache is a thin wrapper around moka that provides typed, TTL-based caching with a unified interface
  • mc-render3d depends on mc-skin for texture loading but handles all 3D geometry, shading, and GPU pipeline setup independently

Benefits of the Workspace Model

Using a Cargo workspace with multiple crates provides several advantages:

Benefit Description
Separation of concerns Each crate owns a single domain, making the codebase easier to navigate and reason about
Parallel compilation Independent crates compile in parallel, speeding up builds
Selective testing Run tests for a single crate with cargo test -p mc-protocol
Clear API boundaries Crate boundaries enforce explicit public APIs via pub visibility
Reusability Crates like mc-protocol could be published independently to crates.io
