Crates Overview

MCInfo-RS is organized as a Cargo workspace consisting of 7 specialized crates. Each crate has a focused responsibility and many can be used independently in your own projects.

Crate Summary

Crate Description Standalone
mc-protocol Custom Minecraft protocol implementation for server pinging (Java, Bedrock, Query) Yes
mc-api Main REST API server built with Axum No (orchestrator)
mc-mojang HTTP client for Mojang's official API Yes
mc-motd MOTD parser and renderer (color codes & Chat Component JSON) Yes
mc-skin Skin downloading, caching, and 2D isometric rendering No (depends on mc-mojang)
mc-render3d Server-side 3D skin rendering via wgpu Yes
mc-cache Generic async TTL cache backed by moka Yes

Dependency Graph

The following diagram shows how the crates relate to each other. mc-api is the top-level binary that depends on all other crates.

mc-api (binary)
  ├── mc-protocol    (server pinging)
  ├── mc-mojang      (player profiles)
  ├── mc-motd        (MOTD parsing)
  ├── mc-skin        (skin handling)
  │   └── mc-mojang
  ├── mc-render3d    (3D rendering)
  └── mc-cache       (TTL caching)

Workspace Configuration

All crates live under crates/ and are declared in the root Cargo.toml:

[workspace]
members = [
    "crates/mc-protocol",
    "crates/mc-api",
    "crates/mc-mojang",
    "crates/mc-motd",
    "crates/mc-skin",
    "crates/mc-render3d",
    "crates/mc-cache",
]
resolver = "2"

Benefits of a Multi-Crate Workspace

  • Separation of concerns — Each crate owns a single domain (protocol, skins, caching, etc.), making the codebase easier to navigate and maintain.
  • Independent reuse — Standalone crates like mc-protocol, mc-mojang, mc-motd, and mc-cache can be pulled into other Rust projects without dragging in the entire API server.
  • Faster incremental builds — Cargo only recompiles crates whose source has changed. Editing a route handler in mc-api won't recompile mc-protocol.
  • Enforced API boundaries — Inter-crate dependencies go through public APIs, preventing accidental tight coupling between subsystems.
  • Parallel compilation — Independent crates compile in parallel, taking full advantage of multi-core CPUs.
  • Shared dependency versions — The workspace resolver ensures all crates use compatible dependency versions, avoiding duplicate builds.

Building & Testing

# Build the entire workspace
cargo build

# Build a specific crate
cargo build -p mc-protocol

# Run all tests across the workspace
cargo test

# Run tests for a single crate
cargo test -p mc-cache

# Run the API server
cargo run -p mc-api
