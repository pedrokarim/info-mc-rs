mc-render3d

Server-side 3D skin rendering powered by wgpu. Generates high-quality PNG renders of Minecraft player models with proper lighting, camera framing, and walking poses. Used for OpenGraph preview images and thumbnail generation.

Edition2021 (for wgpu compatibility)
StandaloneYes
Pathcrates/mc-render3d

Source Structure

FilePurpose

Key Exports

render_skin_png

The main entry point. Takes a skin image and render parameters, returns a PNG-encoded byte vector:

use mc_render3d::{render_skin_png, RenderParams};
use image::open;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let skin = open("skin.png")?.to_rgba8();

    let params = RenderParams {
        width: 512,
        height: 832,
        slim: false,          // Classic (Steve) model
        walking: true,        // Walking pose
        ..Default::default()
    };

    let png_bytes = render_skin_png(&skin, &params).await?;
    std::fs::write("render.png", &png_bytes)?;

    Ok(())
}

RenderParams

Configuration for the 3D render:

pub struct RenderParams {
    /// Output image width in pixels.
    pub width: u32,
    /// Output image height in pixels.
    pub height: u32,
    /// Whether the skin uses the slim (Alex) model.
    pub slim: bool,
    /// Apply a walking pose to arms and legs.
    pub walking: bool,
}

impl Default for RenderParams {
    fn default() -> Self {
        Self {
            width: 512,
            height: 832,
            slim: false,
            walking: false,
        }
    }
}

Rendering Pipeline

The renderer follows a standard GPU rendering workflow, executed entirely on the server:

  1. Device initialization — Creates a headless wgpu device and queue (no display surface needed)
  2. Mesh generation — Builds textured box meshes for each body part with correct UVs mapped from the 64x64 skin texture
  3. Overlay layers — Adds separate slightly-larger meshes for hat, jacket, sleeves, and pants overlay layers
  4. Pose application — Rotates arms and legs for the walking pose using transformation matrices
  5. Camera setup — Positions the camera to match NameMC's framing (slight angle, looking at the model center)
  6. Render pass — Executes the GPU render pass with depth testing and alpha blending
  7. Readback — Copies the rendered texture from GPU memory back to CPU and encodes as PNG

Mesh Generation

Each body part is a textured cuboid with 6 faces, generated from the skin's UV map:

// Example: generating the head mesh
let head = BodyPart::new(
    // Position and dimensions (in Minecraft pixel units)
    Vec3::new(0.0, 24.0, 0.0),  // position
    Vec3::new(8.0, 8.0, 8.0),   // size: 8x8x8

    // UV coordinates from the skin texture (u, v, width, height)
    UvFaces {
        front:  UvRect::new(8, 8, 8, 8),
        back:   UvRect::new(24, 8, 8, 8),
        left:   UvRect::new(0, 8, 8, 8),
        right:  UvRect::new(16, 8, 8, 8),
        top:    UvRect::new(8, 0, 8, 8),
        bottom: UvRect::new(16, 0, 8, 8),
    },
);

Use Cases

  • OpenGraph images — Generates player skin previews for link sharing on social media
  • Thumbnails — Creates skin gallery thumbnails for the web frontend
  • Skin history — Renders previous skins for a player's skin change history

Design Notes

  • Uses Edition 2021 instead of 2024 due to wgpu compatibility requirements.
  • The renderer runs headless — no window or display server is required, making it suitable for server-side rendering in containers.
  • Camera framing matches NameMC's style for a familiar look to players.
  • Face winding and UV mapping are carefully handled to ensure correct rendering of all 6 faces on every body part, including the mirrored left/right sides.
  • Overlay layers (hat, jacket, sleeves, pants) are rendered as slightly inflated meshes on top of the base body parts, with transparency support for partially transparent overlays.
  • The GPU adapter selection prefers discrete GPUs but falls back to integrated graphics or software rendering if needed.
