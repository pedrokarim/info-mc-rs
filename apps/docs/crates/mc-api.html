mc-api

The main REST API server that orchestrates all other crates. Built with Axum, it exposes HTTP endpoints for querying Minecraft servers, looking up player profiles, and rendering skins. The API runs on port 3001 by default.

TypeBinary + Library
StandaloneNo (depends on all other crates)
Pathcrates/mc-api
Default Port3001

Source Structure

routes/server.rsServer ping & query endpoints
routes/player.rsPlayer profile & UUID lookup endpoints
routes/render.rsSkin rendering endpoints (2D & 3D)

services/Business logic layer between routes and crates

Running the Server

# Development
cargo run -p mc-api

# Release build
cargo build --release -p mc-api
./target/release/mc-api

The server starts listening on 0.0.0.0:3001. You can verify it is running with:

curl http://localhost:3001/health

Router Setup

Routes are organized by domain and composed using Axum's nested router pattern:

use axum::{Router, routing::get};

fn build_router(state: AppState) -> Router {
    Router::new()
        .nest("/server", server_routes())
        .nest("/player", player_routes())
        .nest("/render", render_routes())
        .route("/health", get(health_check))
        .layer(cors_layer())
        .layer(rate_limit_layer())
        .with_state(state)
}

Application State

The AppState struct holds shared resources that are injected into handlers via Axum's state extraction:

use mc_cache::TtlCache;

#[derive(Clone)]
pub struct AppState {
    pub http_client: reqwest::Client,
    pub server_cache: TtlCache<String, ServerResponse>,
    pub profile_cache: TtlCache<String, PlayerProfile>,
    pub skin_cache: TtlCache<String, Vec<u8>>,
}

Error Handling

All errors are funneled through a unified ApiError type that automatically maps to appropriate HTTP status codes and JSON error bodies:

use axum::response::IntoResponse;

pub enum ApiError {
    NotFound(String),
    BadRequest(String),
    Upstream(String),
    Internal(String),
    RateLimited,
}

impl IntoResponse for ApiError {
    fn into_response(self) -> axum::response::Response {
        // Maps to 404, 400, 502, 500, 429 respectively
        // Returns JSON: { "error": "...", "status": N }
    }
}

Design Notes

  • The API is designed as a thin orchestration layer — all heavy lifting happens in the individual crates.
  • Each route module is self-contained with its own sub-router, making it easy to add or remove endpoint groups.
  • Caching is handled at the service layer using mc-cache, with different TTLs for server pings (short) vs. player profiles (longer).
  • Rate limiting is applied globally via middleware to prevent abuse.
  • CORS is configured to allow browser-based frontends to call the API directly.
