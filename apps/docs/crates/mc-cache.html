mc-cache

A generic async TTL cache built on top of the moka caching library. Provides a simple, type-safe wrapper for caching expensive computations and API responses with automatic time-based expiration.

StandaloneYes
Pathcrates/mc-cache
StructureSingle file: lib.rs

Key Type

use std::time::Duration;

/// A generic async cache with time-to-live expiration.
///
/// Backed by moka's concurrent cache, which is lock-free and
/// suitable for high-concurrency server workloads.
pub struct TtlCache<K, V> {
    inner: moka::future::Cache<K, V>,
}

impl<K, V> TtlCache<K, V>
where
    K: std::hash::Hash + Eq + Send + Sync + Clone + 'static,
    V: Clone + Send + Sync + 'static,
{
    /// Creates a new cache with the given TTL and max capacity.
    pub fn new(ttl: Duration, max_capacity: u64) -> Self;

    /// Returns the cached value for the key, or computes and inserts it.
    pub async fn get_or_insert_with<F, Fut>(&self, key: K, f: F) -> V
    where
        F: FnOnce() -> Fut,
        Fut: std::future::Future<Output = V>;

    /// Like get_or_insert_with, but the closure can return a Result.
    pub async fn get_or_try_insert_with<F, Fut, E>(
        &self, key: K, f: F
    ) -> Result<V, E>
    where
        F: FnOnce() -> Fut,
        Fut: std::future::Future<Output = Result<V, E>>;

    /// Manually removes an entry from the cache.
    pub fn invalidate(&self, key: &K);

    /// Returns the number of entries currently in the cache.
    pub fn entry_count(&self) -> u64;
}

Usage Examples

Basic cache usage

use mc_cache::TtlCache;
use std::time::Duration;

#[tokio::main]
async fn main() {
    // Cache entries expire after 5 minutes, max 10,000 entries
    let cache: TtlCache<String, String> = TtlCache::new(
        Duration::from_secs(300),
        10_000,
    );

    // First call computes the value
    let value = cache.get_or_insert_with("key".to_string(), || async {
        println!("Computing value...");
        "expensive_result".to_string()
    }).await;

    // Second call returns the cached value (no recomputation)
    let cached = cache.get_or_insert_with("key".to_string(), || async {
        println!("This won't print!");
        "won't be used".to_string()
    }).await;

    assert_eq!(value, cached);
    println!("Entries in cache: {}", cache.entry_count());
}

Caching fallible operations

use mc_cache::TtlCache;
use std::time::Duration;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let cache: TtlCache<String, String> = TtlCache::new(
        Duration::from_secs(60),
        1_000,
    );

    // Only caches successful results; errors are not cached
    let result = cache.get_or_try_insert_with(
        "player_uuid".to_string(),
        || async {
            // Simulate an API call that might fail
            reqwest::get("https://api.example.com/profile")
                .await?
                .text()
                .await
        },
    ).await?;

    println!("Profile: {}", result);
    Ok(())
}

Using in application state

use mc_cache::TtlCache;
use std::time::Duration;

#[derive(Clone)]
struct AppState {
    server_cache: TtlCache<String, ServerResponse>,
    profile_cache: TtlCache<String, PlayerProfile>,
}

impl AppState {
    fn new() -> Self {
        Self {
            // Server pings: short TTL (30 seconds)
            server_cache: TtlCache::new(Duration::from_secs(30), 5_000),
            // Player profiles: longer TTL (10 minutes)
            profile_cache: TtlCache::new(Duration::from_secs(600), 50_000),
        }
    }
}

Manual invalidation

use mc_cache::TtlCache;
use std::time::Duration;

#[tokio::main]
async fn main() {
    let cache: TtlCache<String, i32> = TtlCache::new(
        Duration::from_secs(300),
        100,
    );

    cache.get_or_insert_with("counter".to_string(), || async { 42 }).await;
    println!("Before: {}", cache.entry_count()); // 1

    cache.invalidate(&"counter".to_string());
    println!("After: {}", cache.entry_count());  // 0
}

Configuration

The cache accepts two configuration parameters at construction time:

ParameterDescriptionTypical Values
ttlTime-to-live for each entry after insertion30s (server pings), 5-10min (profiles), 1h (skins)
max_capacityMaximum number of entries before eviction1,000 - 100,000 depending on value size

Design Notes

  • The cache is thread-safe and designed for high-concurrency scenarios — moka uses lock-free data structures internally.
  • get_or_try_insert_with ensures that failed lookups are not cached, so transient errors (network timeouts, API rate limits) will be retried on the next request.
  • The Clone bound on values means the cache returns owned copies. For large values, consider wrapping them in Arc to avoid expensive clones.
  • Eviction is handled automatically by moka using a combination of TTL expiration and LRU-like policies when the max capacity is reached.
  • This crate intentionally provides a minimal API surface — just the methods needed by mc-api — rather than exposing the full moka interface.
